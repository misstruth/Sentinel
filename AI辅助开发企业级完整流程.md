## 💡 最佳实践与避坑指南

### 10.1 AI 使用技巧

#### ✅ 高效用法

1. **分阶段提问**
   - ❌ 错误：一次性让 AI 生成整个项目
   - ✅ 正确：按模块逐步生成，每次聚焦一个功能

2. **提供充足上下文**
   ```
   我的项目使用：
   - 三层架构
   - GORM 作为 ORM
   - 统一错误码
   - 日志使用 zap

   请按照这个规范生成代码
   ```

3. **迭代优化**
   - 第一次：生成基础框架
   - 第二次：补充业务逻辑
   - 第三次：优化性能
   - 第四次：完善错误处理

4. **代码审查**
   - 每次生成代码后，让 AI Review 一遍
   - 重点检查：安全性、性能、规范性

#### ❌ 常见误区

1. **过度依赖 AI**
   - AI 生成的代码必须经过测试
   - 核心业务逻辑需要人工审查
   - 不要直接复制粘贴到生产环境

2. **忽略项目规范**
   - AI 可能不遵循团队代码规范
   - 需要提供规范文档给 AI
   - 生成后需要格式化和调整

3. **缺少测试**
   - AI 生成的代码可能有边界情况未覆盖
   - 必须编写测试用例
   - 测试覆盖率要达标

### 10.2 开发规范

#### 后端规范（Go）

1. **分层架构**
   ```
   Handler（API 层）
     ↓ 只做参数绑定和响应
   Service（业务逻辑层）
     ↓ 核心业务逻辑
   DAO（数据访问层）
     ↓ 只做数据库操作
   ```

2. **错误处理**
   ```go
   // ❌ 错误示例
   if err != nil {
       return nil
   }

   // ✅ 正确示例
   if err != nil {
       return fmt.Errorf("查询任务失败: %w", err)
   }
   ```

3. **日志规范**
   ```go
   // 关键操作必须记录日志
   logger.Info("创建任务",
       zap.String("user_id", userID),
       zap.String("task_id", taskID))
   ```

#### 前端规范（Vue3）

1. **组件拆分**
   - 单个组件不超过 300 行
   - 可复用逻辑抽取为 Composables
   - 业务组件和 UI 组件分离

2. **类型定义**
   ```typescript
   // 所有接口响应必须定义类型
   interface TaskResponse {
       id: string
       title: string
       status: TaskStatus
       created_at: string
   }
   ```

3. **错误处理**
   ```typescript
   // 统一错误处理
   try {
       await createTask(data)
       ElMessage.success('创建成功')
   } catch (error) {
       ElMessage.error(error.message || '创建失败')
   }
   ```
